# DECISION SYNTHESIS — COA extraction on RTX 5060 / 3.8GB free

## The reframe that drives everything

The root cause is **not** a model-size problem and **not** an OCR problem. It is a **missing-cell-grid problem**: every current path hands the small LLM flat text and asks it to *infer* which number belongs to which column. Small LLMs fail at that inference. So the highest-leverage moves are the ones that **remove the inference**, not the ones that buy a marginally smarter guesser.

Two independent families converged on the same anti-deceptive primitive without coordinating: **make the wrong answer structurally impossible or structurally detectable**, rather than hoping a better model guesses right. That convergence is the strongest signal in the whole dossier.

## 1. RANKED RECOMMENDATION

### #1 — Prompt-stack on the EXISTING Ollama, but switch the model to `qwen3:4b` (q4_K_M) and add the layered prompt/schema guards
**The combo: `qwen3:4b` GPU-resident + two-step CoT (Markdown-first → per-row JSON) + nullable schema with SKIP/needsReview in the enum + source_text citation guard in TS.**

Why it wins for *this* machine:
- **VRAM: real fit.** ~2.5GB weights + ~0.3GB KV ≈ 2.8GB. This is the only 7B-class-quality option that actually runs on GPU here — and it ends the CPU-fallback 20–40s pain immediately.
- **Directly biases to SKIP.** The single most important finding in the entire dossier: *if SKIP is not in the enum, the model literally cannot abstain.* Adding `SKIP`/`needsReview` as legal enum tokens + making `spec_min`/`spec_max`/`result` nullable + "return null, do not infer" is the cheapest anti-deceptive win available. The `source_text` citation guard then lets deterministic TS reject any value not literally present in the OCR — converting a deceptive PASS/FAIL into an honest SKIP with zero model trust.
- **Lowest integration cost.** No new service. Ollama model swap + schema + prompt + a TS call loop. ~1 day.
- **Measurably better extraction than the 7B it replaces** (StructEval 67.0% vs 59.0%) *and* uses 40% less VRAM. That is a strict Pareto improvement on paper.
- Local, no GT needed.

The one caveat that will bite if ignored: **do NOT set `think=false` with `format=json-schema` on qwen3** — Ollama bug #15260 silently drops the format constraint and the model emits free text. Omit the `think` param entirely, or embed a `reasoning` field in the schema.

### #2 — Two-Model Agreement Gate (`qwen2.5:3b` extract ↔ `qwen2.5:1.5b` or `qwen3:4b` corroborate; SKIP on any field mismatch)
The best *pure abstention* mechanism, and the only one that catches **systematic** column-collapse (logprobs and self-consistency only catch stochastic noise — a model that always borrows column 2 does so confidently every time). Threshold-free, label-free, ~60–100 lines of TS, both models fit GPU sequentially under 3.8GB. AUROC 0.797 for catching wrong answers vs 0.63 for logprob entropy. **Why not #1:** it doesn't *fix* extraction, it only *gates* it — pair it with #1 rather than choosing between them. It also doubles latency. This is the natural Phase-2 layered on top of #1, and it directly productionizes the disagreement signal you already observed in the 16-file sweep.

### #3 — IBM Docling v2 (TableFormerV2) as a structural pre-parser feeding the LLM a typed cell grid
The only path that attacks the root cause at the *source*: emit a typed grid (row_idx, col_idx, `ched` header tokens) so column mapping is **deterministic in TypeScript** before the LLM sees anything — column-collapse becomes impossible at parse time, not guarded-against. TableFormer is CPU-only (146–213MB), so **zero VRAM pressure** on your Ollama/RapidOCR budget. MIT, local, no GT. **Why not #1:** higher integration cost (new parse stage, route table pages through it, rework the parse step), and its 93.6% TEDS is on *digital* tables (PubTabNet/FinTabNet) — unproven on scanned/rotated COA, so it needs a real pilot before trust. This is the highest-ceiling structural fix and the right Phase-3 if prompt+gate doesn't close the gap.

### #4 — Granite-Vision-3.2-2B Q4_K_M (or Qwen2.5-VL-3B) via Ollama — the VLM path
Drop-in Ollama, fits VRAM (~1.55GB file), removes the OCR hop and gives spatial context that *may* reduce transposed/rotated-table errors. **Why only #4:** the research's own warning is decisive — **a VLM does NOT fix the root-cause column-collapse**; you still need the same SKIP discipline and TS guards. It's a lateral move with unproven COA benefit, not a fix. Worth a side-by-side test only after #1 ships.

### Not recommended (explicit no-go for this machine)
- **Any 7B/8B** (qwen2.5:7b, granite3.3:8b, MiniCPM-V 2.6/8B, OlmOCR-2): exceed 3.8GB → CPU fallback. The current pain.
- **Qwen3-VL-4B, Gemma3-4B vision, SmolVLM2-2.2B**: weights look like they fit but per-image runtime activation (1.2–1.5GB) or KV at real context pushes them over 3.8GB → CPU. Optimistic on paper, NO-GO on GPU.
- **Outlines / XGrammar / vLLM**: require ripping out Ollama for a low-VRAM vLLM server. High risk, no content-accuracy benefit (grammar fixes shape, not which number).
- **Camelot/pdfplumber**: text-layer only — your docs are scanned. Inapplicable.
- **Cloud IDP (Affinda/Nanonets/Docsumo/Azure DI)**: privacy downgrade on customer COA data, AND they reproduce the *exact same* column-collapse failure. No.
- **Table Transformer (TATR), Unstructured.io**: silent wrong grids on borderless tables, miscalibrated confidence — the worst outcome for an anti-deceptive system.

## 2. VRAM / ACCURACY CLAIMS THAT LOOK TOO OPTIMISTIC — verify before trusting

1. **`qwen3:4b` "2.8GB total" — trust but verify the KV at real context.** COA OCR text + two-step CoT scratchpad + few-shot examples can push context past 4K. KV grows with context; at 8K you may be near 3.5GB+. **Pilot must measure peak VRAM with the *actual* prompt, not a toy one.**
2. **Docling TableFormer "no VRAM risk, 2–6s CPU/table" — true on VRAM, optimistic on CPU contention.** TableFormer CPU inference will fight Ollama for cores. Without `OMP_NUM_THREADS` tuning, both slow down. And 93.6% TEDS is **digital tables only** — scanned/rotated COA is explicitly unbenchmarked. Verify accuracy on real COA before trusting the headline number.
3. **Granite-Vision / Qwen2.5-VL-3B "fits 3.8GB" — only at downscaled resolution + short context.** The prior project OOM was almost certainly full-res/BF16. Q4 + 1024px images is the claim; **must verify peak VRAM at the image resolution COA pages actually need** (small text may force higher res → more vision tokens → OOM).
4. **MinerU 2.5 VLM "~4–5GB fits 8GB" — does NOT fit YOUR 3.8GB FREE.** The community quote is about an 8GB *card*, not 3.8GB *free*. NO-GO on GPU here. Drop it from consideration despite the "promising" label.
5. **PaddleOCR-VL "3.3GB with FlashAttention2" — fragile.** Without FA2 it OOMs at 40GB (confirmed bug); FA2 setup on Windows + RTX 5060 is non-trivial and 3.3GB leaves zero KV headroom. Marginal at best.
6. **Cross-Model Perplexity AUROC 0.797 — from MMLU/TriviaQA, not COA tables.** The number is real but off-domain. The *mechanism* (disagreement = misread) is sound and matches your own sweep; don't anchor on the exact 0.797 for COA.
7. **PaddleX wireless/borderless model + Windows GPU wheels.** PaddlePaddle GPU wheels are historically Linux-first. **Verify a CUDA 12.x wheel exists and installs on Windows/RTX 5060 before committing** — this can silently fall to CPU.

## 4. HONEST VERDICT — is any of this worth more than "keep 7b + G1 + bias-to-SKIP"?

**Yes — but the worthwhile change is small, and it is NOT "keep the 7B."**

Be skeptical of the temptation to do nothing. The current 7B is the *single thing the whole dossier agrees is the problem*: it doesn't fit GPU (so it's slow), AND it scores *lower* on structured output (59% StructEval) than a 4B that *does* fit (67%). Keeping it means paying CPU-fallback latency to run the *worst-scoring* viable model. That's the one move with no defense.

So the right framing isn't "big change vs incremental guard" — it's:

- **Incremental guards (your G1 + bias-to-SKIP + SKIP-in-enum + source_text) are 80% of the value and should ship regardless of model.** These are model-independent and directly attack deceptive verdicts. Do them now.
- **The one structural change worth bundling in is the cheap model swap 7B→`qwen3:4b`**, because it is *also* incremental (same Ollama, same pipeline shape, ~1 day) yet simultaneously fixes latency AND raises the accuracy ceiling. It is not a "bigger change" — it's the same size of change as adding a guard, with a strictly better model.

**What is NOT worth it right now:** Docling/MinerU/VLM swaps. They have higher ceilings but cost days of integration and carry unverified-on-COA accuracy. Defer them. They become worth it *only if* the pilot shows the prompt-stack + 4B still leaks confident-wrong verdicts that the source_text guard and two-model gate can't catch — at which point Docling's deterministic typed grid (#3) is the principled next step.

**Bottom line ranking of effort-vs-payoff:**
1. **Ship now (hours–1 day):** SKIP-in-enum + nullable fields + source_text TS guard + two-step CoT + few-shot section-header example **on `qwen3:4b`** (swap off the 7B). This is the recommendation.
2. **Phase 2 (1–2 days) if #1 still leaks:** add the two-model agreement gate to force SKIP on disagreement.
3. **Phase 3 (days) only if structural collapse persists:** Docling TableFormer typed-grid pre-parser, shrinking the LLM's job to single-bound spec-direction classification on locked columns.

The deceptive-verdict problem is solved primarily by **#1's guards**, not by any model. The model swap is worth bundling only because it's free latency + accuracy on the side. Anything heavier should wait for the pilot to prove it's needed.
